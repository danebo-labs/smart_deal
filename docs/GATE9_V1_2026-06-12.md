## B.2 — distinct stop-work control records (`72acf2d`)

Root cause: `BatchResultsParserService#field_record_id` omitted `sw`, so two
emergency-stop records for distinct control stations (platform vs ground)
collapsed to the same `RECORD_ID`, conflicted, and were both discarded. Fix:
append `stop_trigger`/`stop_action` to the fingerprint **only when present**
(historical IDs for every other record type unchanged). Prompt: emit one
separate `STOP_WORK_CONDITION` per documented control station. Shadow `$0.738903`
improved coverage but did not close V1.

## B.3 — recoverable JSON quote repair (`dbe4714`)

Root cause: a page ended `end_turn` with valid-looking JSON containing literal
double quotes inside `chunks[].text` (e.g. `Consulte la sección "Etiquetas"`),
which `JSON.parse` rejected. Fix: new `LlmJsonParser`, strict-first, with a
bounded repair that only escapes quotes that cannot legally close the current
JSON string (driven by an object/array key/value state stack). It does **not**
synthesize keys, braces, commas, or values. Wired into the shared retry
parseability check, `ChunkMergerService`, `BatchResultsParserService`, and
`FieldPhotoResultsParser`. Shadow `$0.813917`: JSON/retries OK; V1 still red
because stop-work scores were <0.6 even though the platform record appeared.

## B.4 — weighted stop-work evidence matching (`34b86d0`)

Root cause: the local evaluator scored stop-work with flat token Jaccard, so a
correct platform record scored below `0.6`. Fix in `Gate9V1Validation`: a
weighted similarity for `STOP_WORK_CONDITION` (`stop_trigger` 0.25, `stop_action`
0.25, `evidence` 0.25, `expected_result`/`action` 0.10, `source` 0.05) plus light
Spanish normalization (clitics, participles, `detenida→detener`). This
**hardens** the gate: `STOP_WORK_STATION_MISMATCH_CAP = 0.55` caps
platform↔ground cross-matches **below** the 0.6 mandatory threshold, and
`STOP_WORK_SOURCE_MATCH_BONUS` only rewards an exact source match. **No
threshold was lowered.** Shadow `$0.825484`: V1 red because the LLM omitted the
slope/gradient stop-work.

## B.5 — conditional operating-limit as STOP_WORK (`2677cdf`)

Root cause: slope/gradient limits were emitted as safety warnings, not
`STOP_WORK_CONDITION`. Fix: prompt now classifies conditional operating limits
with a mandatory recovery action as `STOP_WORK_CONDITION`; a deterministic
generic guardrail in the parser (`CONDITIONAL_TRANSPORT_STOP_REGEX`: "si X
supera/excede … la máquina/plataforma/equipo debe levantarse/transportarse/no
operar…") adds the record when the visible chunk states it. No page/manual/
`record_id` hardcoding. Shadow `$0.764423` failed before quality with
`ParseError: Missing ev in chunk 30 field_record 2`; all results were `end_turn`
(not truncation).

## B.5.1 — quarantine unverifiable field records (`91f9200`)

Root cause: the LLM emitted a non-critical `FUNCTIONAL_TEST` without `ev`
(`r=DATA_NOT_AVAILABLE`, `u` noting the result continues on the next page), and
the parser failed the **entire manual** over one unverifiable ledger record.
Fix:

- `BatchResultsParserService#discard_unverifiable_field_records!` drops **only**
  non-`STOP_WORK` field records without verifiable evidence; `STOP_WORK_CONDITION`
  without evidence still hard-fails.
- Prompt: never emit a `field_record` without `ev`; if no visible quote supports
  it, omit the record and keep the uncertainty in text.
- `Gate9V1Validation`: avoids cross-matching similar stop-work from distinct
  stations (platform vs ground/lower cap) with an exact-source bonus.
