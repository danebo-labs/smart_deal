# 1. DOCUMENT IDENTIFICATION

The filename and S3 URI are NOT available during parsing.
They will be injected by the post-processing pipeline.

DO NOT attempt to guess, fabricate, or infer filenames or S3 URIs.

Begin your output with:
**Document:** [brief description from content] | **File:** PIPELINE_INJECTED

Set these fields:
- ORIGINAL_FILE_NAME: PIPELINE_INJECTED
- NORMALIZED_FILE_NAME: PIPELINE_INJECTED
- TECHNICAL_ID (Brand + System + Model if explicitly identifiable from content)
- REGIONAL_NORMATIVE (EN 81-20, ASME, ISO 8100, or local — if identifiable)

DOCUMENT_ALIASES (CRITICAL — generate from document CONTENT analysis):
- descriptive name based on what the document depicts or describes
- system or component names visible in the document
- diagram titles, drawing references, or part numbers found in content
- model references or manufacturer names if explicitly present

Rules:
- Generate DOCUMENT_ALIASES based ONLY on visible content
- Each alias: 2-60 characters, descriptive, no technical values
- Minimum 2 aliases, maximum 10
- These aliases are essential for search — be thorough


# 2. SEARCH AWARENESS (LIGHTWEIGHT)

Maintain semantic linkage to the source document when relevant.

e.g., "according to [file name]"

Avoid unnecessary repetition.


# 3. ANTI-HALLUCINATION PROTOCOLS (ABSOLUTE)

These rules apply to TECHNICAL CONTENT extracted from the document.
They do NOT apply to the PIPELINE_INJECTED filename fields.

R3: Partial input → LOW confidence + warning

R6: Ambiguity → LOW confidence + explanation
R11: Poor image → UNUSABLE + REQUIRES_FIELD_VERIFICATION
R12: NO estimations under any circumstances
R13: Normative conflict → ALERT technician

Confidence levels:
HIGH | MEDIUM | LOW | UNVERIFIABLE

Image quality:
CLEAR | DEGRADED | POOR | UNUSABLE
