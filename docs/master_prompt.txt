The filename and S3 URI are NOT available during parsing.
They will be injected by the post-processing pipeline.

DO NOT attempt to guess, fabricate, or infer filenames or S3 URIs.

Begin your output with:
**Document:** [brief description from content] | **File:** PIPELINE_INJECTED

Set these fields:
- ORIGINAL_FILE_NAME: PIPELINE_INJECTED

- REGIONAL_NORMATIVE (EN 81-20, ASME, ISO 8100, or local — if identifiable)

DOCUMENT_ALIASES (CRITICAL — generate from document CONTENT analysis):
- descriptive name based on what the document depicts or describes
- system or component names visible in the document
- diagram titles, drawing references, or part numbers found in content
- model references or manufacturer names if explicitly present

Rules:
- Generate DOCUMENT_ALIASES based ONLY on visible content
- Each alias: 2-60 characters, descriptive, no technical values
- Minimum 2 aliases, maximum 10
- These aliases are essential for search — be thorough

# 5A. FIELD-SAFETY EVIDENCE RECORDS

Extract one independently verifiable FIELD_RECORD for each explicit maintenance,
inspection, certification, functional test, troubleshooting, fault, repair, stop-work,
emergency/rescue, installation, commissioning, modernization, schematic-label, safety,
or documentation unit.

The Claude API contract uses compact keys to reduce output tokens:

* k: record type
* h: visible section, heading, figure, or table; DATA_NOT_AVAILABLE when absent
* a: exact action, check, test, condition, or literal schematic label
* r: exact expected result; DATA_NOT_AVAILABLE when the source provides none
* ev: exact supporting phrase, at most 16 source words
* x: optional semicolon-delimited explicit details
* sw: optional [trigger, required action] pair, only for STOP_WORK_CONDITION
* ra: optional explicit repair/reset authority
* u: optional LOW/UNVERIFIABLE/REQUIRES_FIELD_VERIFICATION reason

Allowed k values:

MAINTENANCE_TASK | INSPECTION_CHECK | CERTIFICATION_REQUIREMENT | FUNCTIONAL_TEST |
TROUBLESHOOTING_STEP | FAULT_CONDITION | REPAIR_ACTION | STOP_WORK_CONDITION |
EMERGENCY_OR_RESCUE | INSTALLATION_STEP | COMMISSIONING_STEP |
MODERNIZATION_STEP | SCHEMATIC_LABEL | SAFETY_WARNING | DOCUMENTATION_REQUIREMENT

Use x only for documented role, scope, preconditions, pass/fail criteria,
measurements/limits, tools/PPE, documentation output, function, connection, or value.
Label details compactly, for example:
role=technician; scope=door operator; limit=24 V; output=inspection report

Rules:

* Keep records in the same semantic chunk as their evidence.
* Do not create one-sentence chunks or duplicate a complete procedure in narrative.
* Omit optional keys and unavailable details; do not output empty placeholders.
* Never generate RECORD_ID. Rails creates a deterministic ID after validation.
* Never merge opposing states or independently testable results into one record.
* Keep preparation with a unless it has its own independently verifiable result.
* An inspection without acceptance criteria uses r=DATA_NOT_AVAILABLE.
* STOP_WORK_CONDITION requires both exact sw elements from the same visible fragment.
* A precaution without an explicit stop/prohibit/mark instruction is not stop-work.
* SCHEMATIC_LABEL preserves the literal label; undocumented function/value/connection
  remain absent rather than inferred.
* Preserve terms, modality, codes, labels, units, sequence numbers, and evidence.
* Missing or illegible safety evidence is marked, never completed from assumptions.
